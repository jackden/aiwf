# AIWF v1.7.7

## Release Type
Repository upgrade mechanism release.

## Baseline
Follow-up release after the v1.7.5.post5 metadata-show patch, focused on bootstrap-capable upgrade flows for older repos.

## Included
- Bootstrap-aware `./aiwf` fallback when the canonical runtime is missing.
- `upgrade --check`, `upgrade --dry-run`, and `upgrade --apply`.
- Source package validation before any upgrade action.
- Preservation of `.aiwf/records/**`, `.aiwf/events/**`, `.aiwf/migrations/**`, and config content.
- Legacy layout relocation integration for upgrade apply flows.
- Compatibility shim semantics alignment so `tools/ai_workflow.py` does not trigger relocation when the canonical runtime already exists.
- Upgrade planning now copies `tools/ai_workflow.py` from source as a shim instead of relocating the target repo's legacy tools runtime into `.aiwf/bin/ai_workflow.py`.
- Upgrade documentation added under `.aiwf/docs/upgrading.md` and `.aiwf/docs/examples/upgrade_v1_7_5_post2_to_v1_7_7.md`.
- Validated legacy repo upgrade case from `1.7.5.post2` to `1.7.7`.
- Upgrade report generation under `.aiwf/migrations/`.
- Tool and workflow protocol version alignment to `1.7.7`.

## Compatibility
- No task front matter schema change.
- No event schema change.
- No finalize semantic change.
- Legacy repos should run `upgrade --check` before `upgrade --apply`.

## Version Metadata Policy
For the current lightweight AIWF project, release version, tool version, and workflow protocol version are aligned at `1.7.7`.

The new upgrade mechanism is additive and does not imply a package manager, database migration framework, or silent overwrite of workflow evidence.

## Deferred / Non-goals
- Full install/migrate/uninstall lifecycle remains deferred.
- Rollback engine is not part of this release.
- Network download or auto-upgrade without a source repo remains out of scope.
