# AIWF v1.7.8

## Release Type
Runtime validation hardening patch.

## Summary
AIWF v1.7.8 centralizes CLI date validation across commands that accept `--date`.
Malformed date strings now fail consistently with `AIWF-DATE-002`, invalid calendar dates fail with `AIWF-DATE-003`, `new-task` keeps the non-today date guard, and `backfill` remains exempt from the non-today guard while still validating date format and calendar validity.

## Changed
- Centralized shared date validation for CLI `--date` arguments.
- Applied the shared parser to `next-id`, `list`, `new-task`, and `backfill`.
- Added `AIWF-DATE-002` for malformed date format.
- Added `AIWF-DATE-003` for invalid calendar dates.
- Preserved `new-task` non-today date protection.
- Preserved `backfill` historical-date behavior.
- Added regression coverage for malformed dates, invalid calendar dates, non-today new-task handling, and backfill behavior.
- Added a reusable bug note for the shared date CLI validation gap.

## Compatibility
- No event schema change.
- No workflow protocol semantic upgrade.
- No strict phase state machine.
- No policy-on-path behavior.
- No historical record migration.

## Validation
Targeted regression tests passed for:
- `next-id` malformed date format.
- `next-id` invalid calendar date.
- `list` malformed date format.
- `new-task` invalid calendar date.
- `new-task` non-today rejection without override.
- `new-task` non-today allowance with override.
- `backfill` malformed date rejection.
- `backfill` non-today allowance.
- `list` command filtering behavior.

## Boundary
This release is a runtime hardening patch within the v1.7 series. It does not introduce v1.8 workflow semantics.
