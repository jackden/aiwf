# AIWF v1.7.3

## Release Type
Adoption foundation consolidation release (documentation + wrapper/runtime-facing wording alignment).

## Baseline
Built on the v1.7.x adoption-foundation patch line, with version metadata aligned to `1.7.3`.

## Included
- Repository-local command identity via `./aiwf` with runtime fallback to `.aiwf/bin/ai_workflow.py`.
- Wrapper interpreter hardening (`AIWF_PYTHON`, `.venv/bin/python`, `python3`, `python` with Python >= 3.10 gate).
- Minimal layout-aware records root resolver (`.aiwf/config.yaml` with `layout.records_root`; default remains `docs`).
- AGENTS thin managed block command surface:
  - `./aiwf agents print-block`
  - `./aiwf agents check --path <AGENTS.md>`
  - `./aiwf agents install --path <AGENTS.md> --yes`
- Generic runtime-facing wording cleanup (header/help text).
- Generic generated-template wording cleanup for newly created task/agent/backfill/knowledge artifacts.
- Default pytest collection coverage update (`tests/unit`).
- Wrapper compatibility hardening for cross-platform interpreter selection:
  - checks repo-local `.venv/Scripts/python` and `.venv/Scripts/python.exe` in addition to `.venv/bin/python`
  - prioritizes `python` before `python3` for PATH fallback
  - skips WindowsApps alias interpreters under `Microsoft/WindowsApps`

## Compatibility
- No finalize behavior changes.
- No event schema changes.
- No task metadata schema changes.
- No records-root default change (`docs` remains default).
- No AGENTS managed block semantic changes beyond wording/documentation alignment.
- No protocol semantic changes; this is an adoption compatibility patch for wrapper interpreter selection.

## Version Metadata Policy
For the current lightweight AIWF project, release version, tool version, and workflow protocol version are intentionally aligned at `1.7.3`.

This patch-level metadata alignment does not by itself imply:
- runtime semantic changes
- event schema changes
- finalize gate changes
- diagnostic behavior changes

Version separation remains a future option only when explicit compatibility/runtime channel requirements justify the extra complexity.

## Deferred / Non-goals
- No `init --target` adoption command.
- No manifest-based install/upgrade/migrate/uninstall lifecycle.
- No repo-level governance timeline automation.
- No strict phase-gated finalize state machine.
- No amend-finalized/reopen runtime command.
